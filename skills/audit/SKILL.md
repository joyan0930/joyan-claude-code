# Audit - 完了前監査

`auditor` エージェント（Opus high・読み取り専用）に成果物を独立監査させる。
実装した本人（Executor や自分）の自己申告を信用せず、第三者の目で検査するのが目的。

## 手順

1. **監査スコープを確定する**
   - 引数指定があればそれに従う
   - 省略時: `git status` / `git diff` で未コミット差分を対象にする
     （ブランチ作業中は `git diff main...HEAD` も検討）

2. **監査パッケージを作る**（Auditorは会話を見られない）:
   - **要件**: 元の依頼内容・完了条件を正確に（Auditorは要件充足を判定する）
   - **対象**: 変更ファイル一覧、ブランチ名、diffの取得コマンド
   - **検証手段**: テスト・lint等の実行コマンド
   - **重点観点**: 特に見てほしい点があれば明記（セキュリティ、性能、後方互換など）

3. `auditor` エージェントを Agent ツールで起動する
   - クリティカルな変更（本番影響・セキュリティ関連・データ移行）は
     Agent ツールの `model: "fable"` オーバーライドで監査レベルを上げる

4. **判定処理**:
   - **PASS** → minor発見事項を添えてユーザーに報告
   - **FAIL** → critical/major を修正タスク化してExecutorに委譲（`/advisor-mode` 参照）
     → 修正後、差分を再監査
   - 判定と発見事項は改変せずユーザーに伝える（都合の悪い指摘を省かない）

## 位置づけ

- 既存の専門レビュー（`security-reviewer`, `sql-reviewer`, `terraform-reviewer`,
  `review-team` スキル）と併用可
- `/audit` は「要件充足 + 総合品質」を見る**最終ゲート**。専門観点を深掘りしたい場合は
  専門レビュアーを先に並列実行し、その結果もAuditorへの監査パッケージに含めるとよい
